## 1. 사전 준비

### 요구사항

- Claude Code **v2.1.32** 이상 (`claude --version`으로 확인)
- 실험적 기능이므로 수동 활성화 필요

### 활성화 방법

프로젝트의 `.claude/settings.json`에 추가:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

또는 글로벌 설정 `~/.claude/settings.json`에 추가하면 모든 프로젝트에서 사용 가능.

## 2. 에이전트 팀 vs 서브에이전트

|                | 서브에이전트 (Subagent)                    | 에이전트 팀 (Agent Team)                     |
| :------------- | :----------------------------------------- | :------------------------------------------- |
| **컨텍스트**   | 자체 컨텍스트 윈도우, 결과를 호출자에 반환 | 자체 컨텍스트 윈도우, 완전히 독립적          |
| **커뮤니케이션** | 메인 에이전트에게만 결과 보고             | 팀원끼리 직접 메시지 가능                    |
| **조율**       | 메인 에이전트가 모든 작업 관리             | 공유 태스크 리스트로 자기 조율               |
| **적합한 경우** | 결과만 필요한 집중 작업                    | 토론, 협업이 필요한 복잡한 작업              |
| **토큰 비용**  | 낮음 (결과 요약 후 메인에 반환)            | 높음 (팀원 각각 별도 Claude 인스턴스)        |

### 에이전트 팀이 효과적인 경우

- **리서치/리뷰**: 여러 팀원이 문제의 다른 측면을 동시에 조사
- **새 모듈/기능 개발**: 각 팀원이 서로 다른 파일/모듈을 담당
- **경쟁 가설 디버깅**: 여러 가설을 병렬로 테스트하고 서로 반박
- **크로스 레이어 작업**: 프론트엔드, 백엔드, 테스트를 각각 담당

### 에이전트 팀이 비효율적인 경우

- 순차적 작업
- 같은 파일을 수정하는 작업
- 의존성이 많은 작업
- 단순 루틴 작업 (단일 세션이 더 비용 효율적)

## 3. 팀 시작하기

자연어로 팀 구성을 요청하면 Claude가 팀을 생성한다.

### 기본 예시

```text
CLI 도구를 설계하려고 합니다. 에이전트 팀을 만들어주세요:
- UX 담당 1명
- 기술 아키텍처 담당 1명
- 반대 의견 제시 역할 1명
```

Claude가 자동으로:
1. 팀 생성
2. 팀원 스폰
3. 작업 조율
4. 결과 종합

### 팀원 수와 모델 지정

```text
4명의 팀원으로 팀을 만들어 이 모듈들을 병렬로 리팩토링해주세요.
각 팀원은 Sonnet을 사용하세요.
```

## 4. 디스플레이 모드

### In-process (기본)

- 모든 팀원이 메인 터미널 안에서 실행
- `Shift+Down`으로 팀원 간 이동
- `Enter`로 팀원 세션 보기, `Escape`로 인터럽트
- `Ctrl+T`로 태스크 리스트 토글
- 모든 터미널에서 동작

### Split panes

- 각 팀원이 별도 패인에서 실행
- 클릭으로 팀원 패인에 직접 접근
- **tmux** 또는 **iTerm2** 필요
- VS Code 통합 터미널, Windows Terminal, Ghostty에서는 미지원

### 설정 방법

`~/.claude.json`에서 설정:

```json
{
  "teammateMode": "in-process"
}
```

옵션: `"auto"` (기본값), `"in-process"`, `"tmux"`

단일 세션에서만 변경:

```bash
claude --teammate-mode in-process
```

## 5. 팀 제어하기

### 팀원에게 직접 메시지

- **In-process**: `Shift+Down`으로 팀원 선택 후 메시지 입력
- **Split panes**: 해당 팀원의 패인 클릭 후 직접 상호작용

### 플랜 승인 요구

팀원이 구현 전에 계획을 먼저 세우도록 강제:

```text
인증 모듈 리팩토링을 위한 아키텍트 팀원을 스폰하세요.
변경 전에 플랜 승인을 요구하세요.
```

- 팀원이 계획 완료 시 리드에게 승인 요청 전송
- 리드가 승인하면 구현 시작, 거절하면 피드백을 반영하여 재계획
- 리드의 판단 기준 지정 가능: "테스트 커버리지가 포함된 계획만 승인하세요"

### 태스크 할당 및 클레임

태스크 상태: **pending** → **in progress** → **completed**

- **리드 할당**: 리드에게 특정 팀원에 특정 태스크 할당 요청
- **셀프 클레임**: 팀원이 태스크 완료 후 자동으로 다음 미할당 태스크 선택
- 태스크 간 의존성 자동 관리 (선행 태스크 완료 시 후행 태스크 언블록)
- 파일 락킹으로 동시 클레임 레이스 컨디션 방지

### 팀원 종료

```text
리서처 팀원에게 종료를 요청하세요
```

### 팀 정리

```text
팀을 정리해주세요
```

> **중요**: 반드시 리드를 통해 정리할 것. 팀원이 정리하면 리소스가 불일치 상태가 될 수 있음.
> 활성 팀원이 남아있으면 정리 실패하므로, 팀원을 먼저 종료해야 함.

## 6. 아키텍처

### 구성 요소

| 컴포넌트     | 역할                                                   |
| :----------- | :----------------------------------------------------- |
| **Team Lead** | 팀 생성, 팀원 스폰, 작업 조율을 담당하는 메인 세션    |
| **Teammates** | 각각 할당된 태스크를 수행하는 별도 Claude Code 인스턴스 |
| **Task List** | 팀원들이 클레임하고 완료하는 공유 작업 리스트          |
| **Mailbox**   | 에이전트 간 통신을 위한 메시징 시스템                  |

### 저장 위치

- **팀 설정**: `~/.claude/teams/{team-name}/config.json`
- **태스크 리스트**: `~/.claude/tasks/{team-name}/`

> `config.json`은 런타임 상태(세션 ID, tmux 패인 ID)를 담고 있으므로 수동 편집하지 말 것. 상태 업데이트 시 덮어씌워짐.

### 팀 설정 구조

`config.json`의 `members` 배열에 각 팀원의 이름, 에이전트 ID, 에이전트 타입이 포함됨. 팀원들은 이 파일을 읽어 다른 팀원을 발견할 수 있음.

> 프로젝트 레벨의 팀 설정 파일(예: `.claude/teams/teams.json`)은 인식되지 않음.

## 7. 서브에이전트 정의를 팀원으로 사용

미리 정의한 서브에이전트 타입을 팀원으로 재사용할 수 있다.

```text
security-reviewer 에이전트 타입을 사용하여 인증 모듈을 감사할 팀원을 스폰하세요.
```

적용 사항:
- 해당 정의의 `tools` 허용 목록과 `model`을 따름
- 정의의 본문은 팀원 시스템 프롬프트에 추가 지침으로 부착 (대체가 아님)
- `SendMessage`, 태스크 관리 도구는 `tools` 제한과 무관하게 항상 사용 가능

> **참고**: 서브에이전트 정의의 `skills`와 `mcpServers` 필드는 팀원으로 실행될 때 적용되지 않음. 팀원은 프로젝트/사용자 설정에서 스킬과 MCP 서버를 로드함.

## 8. 권한 및 컨텍스트

### 권한

- 팀원은 리드의 권한 설정으로 시작
- 리드가 `--dangerously-skip-permissions`면 모든 팀원도 동일
- 스폰 후 개별 팀원의 모드 변경 가능 (스폰 시점에는 불가)

### 컨텍스트

- 각 팀원은 자체 컨텍스트 윈도우 보유
- 스폰 시 일반 세션과 동일한 프로젝트 컨텍스트 로드 (CLAUDE.md, MCP 서버, 스킬)
- **리드의 대화 히스토리는 전달되지 않음** → 스폰 프롬프트에 충분한 정보 포함 필요

### 커뮤니케이션

- **자동 메시지 전달**: 팀원이 보낸 메시지는 수신자에게 자동 전달
- **유휴 알림**: 팀원이 작업 완료 후 리드에 자동 알림
- **공유 태스크 리스트**: 모든 에이전트가 태스크 상태 확인 가능
- **message**: 특정 팀원에게 1:1 메시지
- **broadcast**: 모든 팀원에게 동시 전송 (토큰 비용 주의)

> 리드가 팀원에 이름을 부여하며, 이름으로 메시지 전송 가능. 예측 가능한 이름이 필요하면 스폰 시 지정.

## 9. 훅(Hooks)으로 품질 관리

`settings.json`에 훅을 추가하여 팀 운영에 규칙을 적용할 수 있다.

| 훅 이벤트         | 트리거 시점                    | exit code 2 시 동작            |
| :----------------- | :----------------------------- | :----------------------------- |
| `TeammateIdle`     | 팀원이 유휴 상태가 되려 할 때  | 피드백 전송, 팀원 계속 작업    |
| `TaskCreated`      | 태스크가 생성될 때             | 생성 방지, 피드백 전송         |
| `TaskCompleted`    | 태스크가 완료 표시될 때        | 완료 방지, 피드백 전송         |

## 10. 실전 예제

### 병렬 코드 리뷰

```text
에이전트 팀을 만들어 PR #142를 리뷰해주세요. 3명의 리뷰어를 스폰하세요:
- 보안 관련 이슈 집중
- 성능 영향 체크
- 테스트 커버리지 검증
각자 리뷰 후 결과를 보고하도록 하세요.
```

### 경쟁 가설 디버깅

```text
사용자들이 앱이 메시지 하나 보낸 후 종료된다고 보고합니다.
5명의 에이전트 팀원을 스폰하여 서로 다른 가설을 조사하세요.
서로 대화하며 상대방의 이론을 반증하도록 과학적 토론을 시키세요.
합의된 결과를 findings 문서에 업데이트하세요.
```

### 새 기능 병렬 개발

```text
에이전트 팀을 만들어 인증 시스템을 구현하세요:
- 팀원 1: JWT 토큰 생성/검증 모듈 (src/auth/token.ts)
- 팀원 2: 세션 관리 미들웨어 (src/middleware/session.ts)
- 팀원 3: 관련 테스트 작성 (src/__tests__/auth/)
각 팀원은 다른 파일을 담당하도록 하세요.
```

## 11. 베스트 프랙티스

### 1. 팀원에게 충분한 컨텍스트 제공

팀원은 CLAUDE.md 등 프로젝트 컨텍스트는 자동 로드하지만, 리드의 대화 히스토리는 받지 않음. 스폰 프롬프트에 구체적 정보를 포함:

```text
보안 리뷰 팀원을 스폰하세요. 프롬프트:
"src/auth/ 의 인증 모듈을 보안 취약점 관점에서 리뷰하세요.
토큰 처리, 세션 관리, 입력 검증에 집중하세요.
앱은 httpOnly 쿠키에 저장된 JWT 토큰을 사용합니다.
심각도 등급과 함께 이슈를 보고하세요."
```

### 2. 적절한 팀 크기 선택

- **3~5명**이 대부분의 워크플로에 적합
- 팀원당 **5~6개 태스크**가 적절
- 15개 독립 태스크라면 3명의 팀원이 적당
- 3명의 집중된 팀원이 5명의 산만한 팀원보다 나음

### 3. 태스크 크기 적정화

- **너무 작으면**: 조율 오버헤드가 이점을 초과
- **너무 크면**: 오래 작업하다 확인 없이 노력 낭비 위험
- **적절한 크기**: 함수, 테스트 파일, 리뷰 등 명확한 결과물이 있는 자립적 단위

### 4. 리드가 직접 작업하지 않도록

리드가 팀원 대기 없이 직접 구현을 시작하면:

```text
팀원들이 작업을 완료할 때까지 기다린 후 진행하세요
```

### 5. 파일 충돌 방지

두 팀원이 같은 파일을 수정하면 덮어쓰기 발생. 각 팀원이 서로 다른 파일 세트를 담당하도록 분배.

### 6. 모니터링과 조향

팀원 진행 상황을 주기적으로 확인하고, 비효율적인 접근은 리다이렉트. 방치 시간이 길수록 낭비 위험 증가.

### 7. 리서치/리뷰부터 시작

에이전트 팀이 처음이라면 코드 작성이 아닌 PR 리뷰, 라이브러리 리서치, 버그 조사 등 명확한 경계가 있는 작업부터 시작.

## 12. 트러블슈팅

### 팀원이 나타나지 않음

- In-process 모드에서는 `Shift+Down`으로 확인
- 태스크가 팀을 만들 만큼 복잡한지 확인
- Split panes 모드면 tmux 설치 확인: `which tmux`
- iTerm2면 `it2` CLI 설치 및 Python API 활성화 확인

### 권한 프롬프트가 너무 많음

팀원 스폰 전에 [permissions 설정](https://code.claude.com/docs/en/permissions)에서 자주 사용하는 작업을 미리 승인.

### 팀원이 에러로 중단됨

`Shift+Down`으로 해당 팀원 확인 후:
- 추가 지시 전달
- 대체 팀원 스폰

### 리드가 작업 완료 전에 종료됨

리드에게 계속 진행하도록 지시. 또는 팀원 완료를 기다리도록 요청.

### 고아 tmux 세션

팀 종료 후 tmux 세션이 남아있으면:

```bash
tmux ls
tmux kill-session -t <session-name>
```

## 13. 제한사항

| 제한사항                     | 설명                                                                                           |
| :--------------------------- | :--------------------------------------------------------------------------------------------- |
| 세션 재개 불가               | `/resume`, `/rewind`가 in-process 팀원을 복원하지 못함. 재개 후 새 팀원 스폰 필요              |
| 태스크 상태 지연             | 팀원이 태스크 완료를 표시하지 않아 후속 태스크가 블록될 수 있음                                |
| 느린 종료                    | 팀원이 현재 요청/도구 호출 완료 후 종료                                                        |
| 세션당 1개 팀                | 리드는 한 번에 하나의 팀만 관리. 새 팀 시작 전 현재 팀 정리 필요                               |
| 중첩 팀 불가                 | 팀원은 자체 팀을 스폰할 수 없음. 리드만 팀 관리 가능                                          |
| 리드 고정                    | 팀 생성 세션이 영구 리드. 리더십 이전 불가                                                     |
| 권한은 스폰 시 설정          | 모든 팀원이 리드 권한으로 시작. 스폰 후 개별 변경은 가능하나 스폰 시 지정 불가                 |
| Split panes 환경 제한        | VS Code 통합 터미널, Windows Terminal, Ghostty에서 미지원. tmux 또는 iTerm2 필요               |

## 빠른 참조 - 프롬프트 템플릿

### 팀 생성

```text
에이전트 팀을 만들어 [작업 설명]. [N]명의 팀원을 스폰하세요:
- 팀원 1: [역할/담당]
- 팀원 2: [역할/담당]
- 팀원 3: [역할/담당]
각 팀원은 [모델]을 사용하세요.
```

### 플랜 승인 필수

```text
[역할] 팀원을 스폰하세요. 변경 전에 플랜 승인을 요구하세요.
[승인 기준 설명]인 계획만 승인하세요.
```

### 리드 대기 지시

```text
팀원들이 작업을 완료할 때까지 기다린 후 진행하세요.
```

### 팀 종료

```text
모든 팀원에게 종료를 요청한 후 팀을 정리해주세요.
```
