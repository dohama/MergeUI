# A. 철학·원칙

## 1. 기본 원칙

- **우리가 파는 건 디자인 템플릿이다.** 디자인이 곧 상품이므로 퀄리티가 사업의 생사를 결정한다.
- **"와, 이런 서비스가 있었구나"** 수준의 고퀄리티가 기준이다. 그냥 괜찮은 수준은 불합격.
- **화면만 꾸미는 게 아니다.** 실제 SaaS/CRM/E-Commerce 등에서 사용하는 데이터와 항목 기반으로 구성한다.
- **개발자가 가져다 바로 쓸 수 있어야 한다.** 5분 내에 본인 데이터에 머지 가능한 구조.

## 2. 작업 순서

1. **디자인 시스템 정의** (B 디자이너 주도) — 폰트/컬러/레이아웃 결정
2. **캡틴 승인**
3. **tokens.css 생성** — 모든 변수 정의
4. **HTML/CSS 구현**
5. **data.js 데이터 분리**
6. **전 에이전트 합동 검수**

## 3. 디자이너(B) 특별 지침

1. **작업 시작 전 반드시 캡틴에게 제안**할 것:
   - 이 테마에 이 폰트가 어울릴 것 같습니다 (이유 포함)
   - 이 컬러 조합을 제안합니다 (메인/서브/서드 + 샘플 이미지)
2. **뻔하지 않게.** 재미있는 요소를 넣을 것.
3. **의견 적극 제시.** 캡틴이 허용했음. 좋은 아이디어가 있으면 말할 것.
4. **검수 시 1px도 놓치지 않을 것.** 텍스트 가독성, 컬러 밸런스, 간격 일관성.

## 4. 전 에이전트 공통 지침

- 모든 테마 작업 마무리 시 **6명 전원 합동 검수** 필수
- 검수 시 **"이걸 보고 내가 $19 낼 의향이 있는가?"** 기준으로 판단
- 심각도는 **1~5 수치**로 표기 (5=즉시수정, 1=안해도됨)
- SoftDesk가 **모든 테마의 기준**. 여기서 확립된 퀄리티/구조를 이후 테마에 적용

# B. 시각 스펙

## 5. 컬러 전략 + 토큰 체계

### 5.1 전략 원칙
- 각 테마별 **배경 톤 1개 고정**. 라이트/다크 전환 기능 넣지 않는다.
- **액센트 컬러 프리셋**을 유동적으로 제공 (테마별 3~5벌, 무료 템플릿은 1개 고정).
- 구독자는 tokens.css 수정으로 자유 커스터마이징 가능.

### 5.2 주의사항
- **컬러 다양성**: AI가 기본으로 보라색을 쓰는 경향이 있으니 항상 다양한 컬러를 캡틴에게 제안할 것.
- 메인 컬러가 텍스트로 사용될 때 **배경 대비 가독성** 반드시 확인 (WCAG AA 4.5:1).
- 같은 컬러로 도배하면 눈아프다. **메인/서브/서드 밸런스** 필수.
- 서브 컬러가 밝으면 본문에서 자제, 그라디언트나 포인트에만 사용.

### 5.3 SoftDesk 예시
- 메인: `#1ACEFF`
- 서브: `#44EEEF` (그라디언트용, 밝은 곳 자제)
- 서드: `#6C5CE7` (밸런스용)

### 5.4 필수 CSS 토큰 (모든 테마 공통)

```css
/* 액센트 */
--merge-color-primary
--merge-color-primary-hover
--merge-color-primary-light
--merge-color-primary-dark

/* 배경 */
--merge-bg-base          /* 페이지 배경 */
--merge-bg-surface       /* 카드/위젯 배경 */
--merge-bg-elevated      /* 모달/드롭다운 */
--merge-bg-inset         /* 오목한 요소 */

/* 텍스트 */
--merge-text-primary     /* 제목, 핵심 텍스트 */
--merge-text-secondary   /* 본문 */
--merge-text-muted       /* 라벨, 메타 */
--merge-text-inverse     /* 액센트 배경 위 텍스트 */

/* 상태 */
--merge-status-success / -light
--merge-status-warning / -light
--merge-status-error / -light
--merge-status-info / -light

/* 테두리 */
--merge-border-default   /* 은은한 구분선 */
--merge-border-strong    /* 명확한 테두리 */

/* 차트 팔레트 */
--merge-chart-1 ~ --merge-chart-8
```

### 5.5 차트 팔레트 규칙
- 최대 8색
- 색약 대응 (deuteranopia-safe) — 빨강/초록 동시 사용 시 형태/패턴으로 구분
- 인접 색상 간 명도 차이 확보

### 5.6 액센트 프리셋 (모든 테마에 4벌 제공)
- `variables.css`에 주석 처리된 프리셋으로 포함
- 구독자가 주석 해제만으로 전환 가능

## 6. 폰트 정책 + 스케일

### 6.1 원칙
- **저작권 무료 폰트만** 사용 (Google Fonts OFL 라이선스).
- 작업 시작 전 디자이너(B)가 **테마 컨셉에 맞는 폰트를 캡틴에게 제안**할 것.
- "Inter만 쓰면 됩니다"는 금지.

### 6.2 추천 폰트 풀

| 폰트 | 특징 | 잘 어울리는 테마 |
|------|------|----------------|
| **Plus Jakarta Sans** | 따뜻하고 부드러움 | 뉴모피즘, 친근한 SaaS |
| **DM Sans** | 모던 지오메트릭 | 미니멀, 핀테크 |
| **Inter** | 가장 무난, 가독성 좋음 | 범용 (영어 대시보드 1위) |
| **Manrope** | 기하학적 + 읽기 편함 | 데이터 중심 |
| **Outfit** | 깨끗하고 현대적 | 트렌디한 SaaS |

**Inter 선정 근거 (기본 폴백)**:
- x-height가 높아 작은 크기에서도 가독성 우수
- 유사 글자(I/l/1, O/0) 명확 구분
- `tabular-nums` 지원 — 데이터 테이블에서 숫자 열 정렬 보장
- Variable Font 지원 — 파일 1개로 모든 굵기 사용, 용량 절약

### 6.3 폰트 스케일

| 요소 | Size | Weight | Line Height | Letter Spacing | 용도 |
|------|------|--------|-------------|----------------|------|
| H1 | 32px | 700 | 1.25 (40px) | -0.02em | 페이지 타이틀 |
| H2 | 24px | 600 | 1.33 (32px) | -0.01em | 섹션 타이틀 |
| H3 | 20px | 600 | 1.4 (28px) | -0.01em | 카드/위젯 타이틀 |
| Body | 16px | 400 | 1.5 (24px) | 0 | 본문 텍스트 |
| Caption | 13px | 500 | 1.54 (20px) | 0.01em | 라벨, 보조 텍스트 |
| Small | 12px | 400 | 1.5 (18px) | 0.02em | 메타 정보 (절대 최소) |

**근거**:
- 본문 16px — Material Design 3, Apple HIG 권장 최소값
- line-height 1.5 — WCAG 2.2 텍스트 간격 기준 (폰트 크기의 1.5배 이상)
- 12px 미만 금지 — 접근성 업계 최소 기준
- letter-spacing는 큰 크기에서 타이트하게(-0.02em), 작은 크기에서 넓게(+0.02em)

### 6.4 숫자 표시 규칙

```css
/* 데이터 테이블, 통계 카드 등 숫자 영역에 필수 적용 */
font-variant-numeric: tabular-nums lining-nums;
```

- `tabular-nums`: 모든 숫자가 동일 너비 → 열 정렬 보장
- `lining-nums`: 숫자가 베이스라인에 정렬 → 테이블에 적합

### 6.5 보조 폰트: JetBrains Mono
- 코드 블록, 라이선스 키, API 키 등 모노스페이스 필요 시
- 13px / 400 / line-height 1.54

## 7. 접근성 (WCAG 2.2)

### 7.1 색상 대비 — 필수

| 대상 | AA 기준 | AAA 기준 |
|------|---------|----------|
| 일반 텍스트 (< 18px bold, < 24px) | **4.5:1** | 7:1 |
| 큰 텍스트 (≥ 18px bold 또는 ≥ 24px) | **3:1** | 4.5:1 |
| UI 컴포넌트 (버튼 테두리, 입력 필드 등) | **3:1** | — |

**MergeUi 목표: AA 필수, AAA 권장**

### 7.2 최소 사이즈

| 대상 | 최소값 | 근거 |
|------|--------|------|
| 본문 폰트 | **16px** | Material Design 3, Apple HIG |
| 절대 최소 폰트 | **12px** | 업계 합의 최소선 |
| 터치 타겟 (모바일) | **44×44px** | WCAG 2.2 Level AA |
| 클릭 타겟 (데스크톱) | **32×32px** | 마우스 정밀도 고려 |

### 7.3 포커스 인디케이터

```css
/* 모든 인터랙티브 요소에 적용 */
:focus-visible {
  outline: 2px solid var(--merge-color-primary);
  outline-offset: 2px;
}
```

- 두께: 최소 2px
- 포커스/비포커스 간 대비: 3:1 이상
- `outline-offset`으로 요소와 간격 확보

### 7.4 키보드 네비게이션 — 필수

| 요구사항 | 구현 방법 |
|----------|----------|
| 모든 인터랙티브 요소 Tab 접근 | `tabindex` 순서 보장 |
| 논리적 Tab 순서 | DOM 순서 = 시각 순서 |
| Enter/Space로 활성화 | 버튼, 링크, 토글 등 |
| Escape로 닫기 | 모달, 드롭다운, 사이드 패널 |
| 화살표 키 네비게이션 | 탭, 메뉴, 라디오 그룹 내부 |

### 7.5 ARIA 필수 요소

| 컴포넌트 | 필수 ARIA |
|----------|----------|
| 사이드바 | `role="navigation"`, `aria-label="Main navigation"` |
| 탑바 | `role="banner"` |
| 메인 콘텐츠 | `role="main"` |
| 차트 | `role="img"`, `aria-label="차트 설명"` |
| 테이블 | `<table>` 시맨틱 사용, `<th scope="col">` |
| 알림/뱃지 | `aria-live="polite"` |
| 토글 | `role="switch"`, `aria-checked` |
| 모달 | `role="dialog"`, `aria-modal="true"`, 포커스 트랩 |

# C. 코드·구조

## 10. 코드 구조 — 개발자 친화

### 10.1 데이터 분리 (필수)

```
templates/테마명/
```

- 시맨틱 태그 필수: `<header>`, `<nav>`, `<main>`, `<section>`, `<aside>`, `<footer>`

- 모든 `<input>`에 `<label>` 연결 또는 `aria-label`

### 10.6 CSS 규칙
- 모든 값은 CSS 변수(`--merge-*` 또는 테마별 prefix)로 정의
- 변수명 규칙: `--{prefix}-{카테고리}-{속성}` (예: `--sd-bg-surface`)

- 미디어 쿼리: 모바일 퍼스트 아님, 데스크톱 기본 + 축소 방향

- 최소한의 바닐라 JS만 사용
- 프레임워크 의존성 없음 (Chart.js만 예외)
- 이벤트 위임 활용
- `DOMContentLoaded` 후 초기화

## 11. 국제화 (i18n·RTL)

### 11.1 RTL 대응 — CSS Logical Properties 사용

```css
/* ✅ 올바른 방법 (RTL 자동 대응) */
margin-inline-start: 16px;
padding-inline-end: 8px;
text-align: start;

/* ❌ 피할 방법 (RTL에서 수동 수정 필요) */
margin-left: 16px;
padding-right: 8px;
text-align: left;
```

처음부터 Logical Properties를 쓰면 RTL 대응 비용이 거의 제로.

### 11.2 날짜/숫자 포맷
- 하드코딩 금지 — `Intl.DateTimeFormat()`, `Intl.NumberFormat()` 사용
- 데모 데이터는 ISO 형식(YYYY-MM-DD) 또는 미국 형식(MMM DD, YYYY) 사용

### 11.3 텍스트 길이 변동 대응
- 번역 시 텍스트 길이 30~200% 변동 가능
- 고정 너비 레이아웃 피하기
- `text-overflow: ellipsis` + 툴팁 조합 권장

## 13. 보안 — 코드 보호

### 13.1 홈페이지에서 보여줄 때
- 테마 프리뷰는 **iframe**으로 보여줌 (경쟁사 표준)
- iframe 내 코드에 **난독화(obfuscation)** 적용 검토
- **우클릭 방지·드래그 방지**는 UX 해치므로 하지 않음
- 핵심 방어: "50개 파일을 하나하나 복사하는 것보다 $19 내고 zip 받는 게 빠르다"는 가치 제공

### 13.2 다운로드 패키지
- Pro 사용자만 다운로드 가능 (서버에서 권한 확인 — `hasActiveSubscription` 체크)
- 다운로드 시 라이선스 키 포함
- 코드에 라이선스 주석 포함:

```
/* Licensed to: user@email.com | Key: MERGE-PRO-XXXX */
```
